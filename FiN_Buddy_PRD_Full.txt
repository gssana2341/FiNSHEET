────────────────────────────────────────────────────────────────────────────────
  A1. THE BIG IDEA — ทำไม FiN Buddy ถึงต้องมีอยู่
────────────────────────────────────────────────────────────────────────────────

  ปัญหาที่เราแก้:
  เด็กไทยมีเลคเชอร์เยอะ มีข้อสอบเยอะ แต่ไม่มีเวลาอ่านทั้งหมด
  และตลาดซื้อขายสรุปที่มีอยู่ในปัจจุบันยังไม่แฟร์กับ Creator
  แพลตฟอร์มหัก GP สูง Creator ต้องขายแพง นักเรียนซื้อแพง ทุกคนแพ้

  วิธีที่เราแก้:
  1. AI ที่เข้าใจบริบทการเรียนไทย → ย่อยเลคเชอร์ให้เป็นภาษาเพื่อนติวเพื่อน
  2. Marketplace ที่หัก GP ต่ำที่สุดในตลาด → ทุกฝ่ายได้ประโยชน์
  3. Experience ที่ทำให้รู้สึกอยากอ่าน ไม่ใช่แค่เครื่องมือ

  เป้าหมาย:
  FiN Buddy ต้องเป็น "Space สำหรับคนอยากสอบผ่าน"
  เปิดแอปแล้วได้ Energy กลับไป ไม่ใช่แค่ Tool ที่ใช้แล้วปิด


────────────────────────────────────────────────────────────────────────────────
  A2. THE CORE VIBE — ความรู้สึกที่ User ต้องได้รับ
────────────────────────────────────────────────────────────────────────────────

  One-liner:
  "FiN Buddy คือ Studygram ที่มีชีวิต"

  รายละเอียด:
  - เปิดมาแล้วต้องรู้สึกมีไฟ ไม่กดดัน
  - รู้สึกว่าการสอบผ่านไม่ใช่เรื่องยาก
  - AI ทำให้ข้อมูลยากๆ ย่อยง่ายขึ้น
  - มีสรุปจากคนเก่งทั่วประเทศรออยู่
  - มีเพื่อนที่เรียนด้วยกัน (Leaderboard, Streak)

  Feeling Keywords (ให้ Marketing ใช้เป็น Tone of Voice):
  - กระตุ้น ไม่กดดัน
  - ฉลาด ไม่วิชาการ
  - เพื่อน ไม่อาจารย์
  - สนุก ไม่เบื่อ
  - ได้ผล ไม่ซับซ้อน


────────────────────────────────────────────────────────────────────────────────
  A3. UNIQUE SELLING PROPOSITIONS (USPs)
────────────────────────────────────────────────────────────────────────────────

  USP #1 — AI ที่เหนือกว่า (The Ultimate AI Helper)
  ─────────────────────────────────────────────────
  - ไม่ใช่แค่ Copy-Paste แล้วสรุป
  - AI ถูก Prompt Engineer ให้เข้าใจบริบทการเรียนของเด็กไทย
  - Output ออกมาเป็น "ภาษาเพื่อนติวเพื่อน" อ่านรู้เรื่องทันที
  - รองรับไฟล์ PDF, รูปภาพ (OCR), และ Text โดยตรง
  - Streaming Output แบบ ChatGPT — รู้สึกไม่ต้องรอ

  USP #2 — Marketplace ที่แฟร์ที่สุด (Creator-Friendly)
  ───────────────────────────────────────────────────────
  เราคือ "ตัวกลาง" ที่ทุกฝ่ายได้ประโยชน์:

  | Platform      | GP Cut | ผลลัพธ์                          |
  |---------------|--------|----------------------------------|
  | คู่แข่งทั่วไป  | 30%+   | Creator ขายแพง / User จ่ายแพง   |
  | FiN Buddy     | 10-15% | Creator กำไรดี / User ราคาคุ้ม  |

  กลยุทธ์: Volume-based
  ขายถูกลง → ขายได้มากขึ้น → Creator กำไรรวมมากกว่าเดิม

  USP #3 — Motivation Engine (ระบบกระตุ้นให้อยากเรียน)
  ──────────────────────────────────────────────────────
  - Study Streak: จำนวนวันที่เข้ามาเรียนต่อเนื่อง (เหมือน Duolingo)
  - Leaderboard: อันดับผู้เรียนสัปดาห์นี้
  - Progress Badge: รางวัลเล็กๆ เมื่อสรุปครบ / Quiz ผ่าน
  - Goal Tracker: ตั้งเป้าหมายวันสอบ แอปนับถอยหลังให้

────────────────────────────────────────────────────────────────────────────────
  C1. PROJECT OBJECTIVE & SCOPE
────────────────────────────────────────────────────────────────────────────────

  Objective:
  สร้าง EdTech Platform (MVP) ที่มี 2 Core Feature หลัก คือ
  (1) AI Summarizer และ (2) Marketplace ซื้อขายสรุป
  โดย UX/UI ต้องกระตุ้นให้อยากเรียน และรองรับทุก Device สมบูรณ์

  Tech Stack ที่แนะนำ:
  - Frontend:   React Native (หรือ Flutter หากทีมถนัดกว่า)
  - Backend:    Node.js + Express (หรือ NestJS สำหรับโครงสร้างที่ดีกว่า)
  - Database:   PostgreSQL (Relational) + Redis (Cache / Session)
  - Storage:    AWS S3 หรือ Cloudflare R2 สำหรับไฟล์ PDF
  - AI:         OpenAI API (GPT-4o-mini สำหรับ Cost Control) หรือ Gemini Flash
  - OCR:        Google Vision API หรือ Tesseract.js (ถ้าต้องการ On-device)
  - Payment:    Omise (รองรับ PromptPay + บัตรเครดิต ไทย)
  - CDN:        Cloudflare
  - Auth:       Firebase Auth หรือ Supabase Auth (Google / Apple Sign-In)

  Timeline (MVP):
  Week 1-2: Auth + AI Engine + File Upload
  Week 2-3: Marketplace (Buy Flow) + Payment
  Week 3-4: Creator Dashboard + Legal System + Polish

────────────────────────────────────────────────────────────────────────────────
  C3. CORE FEATURE: AI STUDY ENGINE
────────────────────────────────────────────────────────────────────────────────

  C3.1 File Input & Parsing
  ──────────────────────────
  ประเภทไฟล์ที่รองรับ:
  - PDF (หลัก)
  - รูปภาพ: JPG, PNG, HEIC (ต้องผ่าน OCR ก่อน)
  - Text: Paste โดยตรงใน Text Area

  PDF Parser Pipeline (ทำตามลำดับนี้):
  Step 1: Extract raw text จาก PDF (ใช้ pdf-parse หรือ pdfminer)
  Step 2: Clean text
         - ลบ Header / Footer ที่ซ้ำในทุกหน้า
         - แก้ Line Break ที่ผิดเพี้ยน (ขึ้นบรรทัดกลางคำ)
         - ลบ Special Characters ที่เกิดจาก Encoding
  Step 3: Chunk text เป็น Segment ≤ 2,000 tokens ต่อ chunk
  Step 4: ส่งให้ AI พร้อม System Prompt

  OCR Pipeline:
  Step 1: Preprocess รูปภาพ (Deskew, Denoise, Enhance Contrast)
  Step 2: ส่งให้ Google Vision API หรือ Tesseract
  Step 3: ส่ง Output ต่อเข้า PDF Parser Pipeline ข้างต้น

  C3.2 AI Processing — System Prompt (สำคัญมาก)
  ─────────────────────────────────────────────────
  System Prompt ที่ Dev ต้องใช้ (ปรับได้):
  ─────────────────────────────────────────
  "คุณคือเพื่อนที่ฉลาดมากที่กำลังอธิบายเนื้อหาการเรียนให้เพื่อน
   พูดด้วยภาษาไทยที่เข้าใจง่าย เหมือนคุยกัน ไม่ต้องการภาษาวิชาการ
   เมื่อสรุป ให้จัดรูปแบบดังนี้:

   ## ประเด็นหลัก (ไม่เกิน 5 ข้อ)
   ## รายละเอียดสำคัญ
   ## สิ่งที่ออกสอบบ่อย (ถ้ามี)
   ## จำง่ายๆ ด้วยคำย่อ หรือ Trick

   ใช้ Emoji เล็กน้อยเพื่อแยกหัวข้อ ทำให้ดูไม่น่าเบื่อ"
  ─────────────────────────────────────────────

  Model ที่แนะนำ:
  - GPT-4o-mini หรือ Gemini 1.5 Flash (Cost-effective สำหรับ MVP)
  - พิจารณา Claude Haiku สำหรับ OCR + Short Summary

  Token Limit:
  - Input:  ไม่เกิน 4,000 tokens ต่อ Request
  - Output: ไม่เกิน 1,500 tokens ต่อ Request
  - ถ้าไฟล์ใหญ่กว่า: ให้ Chunk แล้ว Summarize เป็นรอบๆ

  C3.3 Streaming Output
  ──────────────────────
  ต้องใช้ Server-Sent Events (SSE) หรือ WebSocket
  เพื่อทยอยส่ง Token ออกมาทีละตัวแบบ ChatGPT

  Code Pattern (Node.js):
  ────────────────────────
  const stream = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    stream: true,
    messages: [...]
  });

  for await (const chunk of stream) {
    const token = chunk.choices[0]?.delta?.content || "";
    res.write(`data: ${JSON.stringify({ token })}\n\n`);
  }
  res.end();
  ────────────────────────

  Frontend ต้องรับ Stream และ append text แบบ Real-time

  C3.4 Post-Summary Actions (CTA Buttons)
  ─────────────────────────────────────────
  หลังจาก AI สรุปเสร็จ ต้องแสดงปุ่มต่อไปนี้ทันที:

  [ 🧪 สร้าง Quiz จากสรุปนี้ ]
     → ส่ง Summary กลับไปให้ AI สร้าง 5-10 ข้อ Multiple Choice

  [ 🃏 แปลงเป็น Flashcards ]
     → AI แยก Key Term : Definition เป็น Pair
     → แสดงใน Flip Card Interface

────────────────────────────────────────────────────────────────────────────────
  E1. DIGITAL WATERMARKING
────────────────────────────────────────────────────────────────────────────────

  เป้าหมาย:
  ทุกครั้งที่ User ซื้อหรือดาวน์โหลดสรุป ระบบต้องฝัง ID เฉพาะของ
  ผู้ซื้อลงไปในไฟล์ PDF เพื่อใช้ตรวจสอบหากมีการนำไปปล่อยเถื่อน

  วิธีทำ (เลือก 1 วิธี ตาม Complexity):

  วิธีที่ 1: Metadata Embedding (ง่ายที่สุด สำหรับ MVP)
  ─────────────────────────────────────────────────────
  - ฝัง User ID ลงใน PDF Metadata (Field: "Author" หรือ Custom Field)
  - ใช้ Library: python-pypdf หรือ pdf-lib (Node.js)
  - ข้อเสีย: ลบออกได้ด้วย Metadata Cleaner

  Code Pattern (Node.js + pdf-lib):
  const { PDFDocument } = require('pdf-lib');
  async function addWatermark(pdfBytes, userId, orderId) {
    const doc = await PDFDocument.load(pdfBytes);
    doc.setAuthor(`FiN-${userId}-${orderId}`);
    doc.setKeywords([`wm:${userId}`, `order:${orderId}`]);
    return await doc.save();
  }

  วิธีที่ 2: Visible Subtle Text Watermark (แนะนำ MVP)
  ──────────────────────────────────────────────────────
  - พิมพ์ข้อความ User ID / Email เล็กๆ มุมล่างขวาทุกหน้า
  - ความโปร่งใส (Opacity): 15-20% สีเทาอ่อน
  - ขนาด Font: 7-8px (เห็นแต่ไม่รบกวนการอ่าน)
  - ข้อดี: มองเห็นได้ถ้าเอาไปโพสต์ในโซเชียล

  วิธีที่ 3: Invisible Steganographic Watermark (สำหรับ V2)
  ───────────────────────────────────────────────────────────
  - ใช้ Library: imatag หรือ Digimarc API
  - ฝัง Data ลงใน Pixel Pattern ของภาพที่ฝังใน PDF
  - ข้อดี: ตรวจสอบได้แม้ถ่ายรูปหน้าจอ
  - ข้อเสีย: ราคาสูง เหมาะกับ Scale ขึ้น

  สำหรับ MVP: ใช้ทั้งวิธีที่ 1 และวิธีที่ 2 รวมกัน


────────────────────────────────────────────────────────────────────────────────
  E2. AI CONTENT MODERATION
────────────────────────────────────────────────────────────────────────────────

  เป้าหมาย:
  ก่อนสรุปจะถูก Publish บน Marketplace ต้องผ่านการตรวจสอบอัตโนมัติ

  Pipeline:
  Step 1: Extract text จาก PDF ที่ Creator อัปโหลด
  Step 2: ส่งให้ AI Model (GPT-4o-mini หรือ Perspective API)
  Step 3: ตรวจสอบตาม Checklist:
    - มีเนื้อหาลามกอนาจารหรือไม่?
    - มี Hate Speech หรือไม่?
    - มีข้อมูลส่วนบุคคลของผู้อื่น (เลขบัตร, เบอร์โทร) หรือไม่?
    - มีการระบุชื่อเจ้าของลิขสิทธิ์ชัดเจนเกินไปหรือไม่?
      (เช่น ระบุว่าคัดลอกมาจากหนังสือของใคร)
  Step 4: Return Score (0-100) + Flag ที่พบ

  Threshold:
  - Score < 30: Publish อัตโนมัติ
  - Score 30-70: ส่ง Admin Queue ให้ตรวจสอบ แต่ Publish ชั่วคราว
  - Score > 70: Reject + แจ้ง Creator พร้อมเหตุผล

  System Prompt สำหรับ Moderation:
  ─────────────────────────────────
  "คุณคือ Content Moderator ของแพลตฟอร์มการศึกษา
   ตรวจสอบเนื้อหาต่อไปนี้และตอบกลับเป็น JSON เท่านั้น:
   {
     score: 0-100,    (0 = ปลอดภัย, 100 = อันตรายมาก)
     flags: [],       (รายการปัญหาที่พบ)
     reason: ""       (อธิบายสั้นๆ ถ้ามีปัญหา)
   }
   ตรวจสอบ: เนื้อหาลามก, Hate Speech, ข้อมูลส่วนตัว, การละเมิดลิขสิทธิ์อย่างชัดเจน"
  ─────────────────────────────────

  POST   /creator/items       → Upload new item
  PATCH  /creator/items/:id   → Edit item

  GET    /creator/dashboard   → Analytics overview
  POST   /creator/withdraw    → Request withdrawal

  Reports:
  POST   /reports             → Submit report
  GET    /admin/reports       → Admin: get report queue
  PATCH  /admin/reports/:id   → Admin: approve/remove

  Payments (Webhook):
  POST   /webhooks/omise      → Receive payment confirmation from Omise


────────────────────────────────────────────────────────────────────────────────
  F4. API SECURITY
────────────────────────────────────────────────────────────────────────────────

  Authentication:
  - ทุก Endpoint (ยกเว้น Auth) ต้องมี JWT Bearer Token
  - Token Expiry: Access Token 15 นาที / Refresh Token 30 วัน

  Rate Limiting:
  - /ai/summarize:            5 requests/min ต่อ user
  - /marketplace/items (GET): 60 requests/min ต่อ IP
  - /auth/*:                  10 requests/min ต่อ IP

  File Security:
  - ไฟล์ PDF ทุกอันเก็บใน Private S3 Bucket
  - ไม่มี Public URL เด็ดขาด
  - ใช้ Signed URL เท่านั้น (Expiry: 1 ชั่วโมง)
  - ทุกครั้งที่ Generate Signed URL ต้องตรวจสอบว่า User เป็นเจ้าของ Order นั้น

  Content Protection:
  - Disable Text Selection ใน PDF Viewer ที่แสดงเนื้อหาที่ซื้อ
    (ใช้ CSS: user-select: none + preventDefault ใน onCopy event)
  - ไม่เปิด External Link ออกจาก App โดยตรง (ป้องกัน Phishing)

  Input Validation:
  - ตรวจสอบ File Type จาก Magic Bytes ไม่ใช่แค่ Extension
  - จำกัดขนาดไฟล์: PDF ≤ 50MB / รูปภาพ ≤ 10MB
  - Sanitize ทุก Text Input ก่อน Query Database

  H3. NON-NEGOTIABLE QUALITY STANDARDS
────────────────────────────────────────────────────────────────────────────────

  Performance:
  - AI Summary เริ่มแสดง Token แรกภายใน 3 วินาที
  - หน้า Feed โหลดภายใน 1.5 วินาที (Cached)

  Reliability:
  - Payment Flow ต้องไม่ Error ไม่ว่ากรณีใดก็ตาม
  - ถ้า AI Error ต้องแสดง Error State ชัดเจน พร้อมปุ่ม "ลองอีกครั้ง"
  - ถ้า Upload ล้มเหลว ต้องคืน Credit ทันที (Rollback)

  UX Standards:
  - Touch Target ทุกปุ่มต้องไม่น้อยกว่า 44x44px
  - Font Size ใน Input ต้องไม่น้อยกว่า 16px (iOS Safari zoom rule)
  - Loading State ทุก Async Action ต้องมี Skeleton หรือ Spinner
  - Empty State ทุกหน้าต้องมี Illustration + ข้อความ + CTA

  1. User สามารถ Sign In → Upload PDF → ได้ AI Summary ที่อ่านรู้เรื่อง
  2. User สามารถ Browse Marketplace → Preview → ซื้อด้วย Credit
  3. Creator สามารถ Upload สรุป → ระบบ Moderate → Publish
  4. ทุก PDF ที่ซื้อมี Watermark ของผู้ซื้อ
  5. แอปทำงานได้ดีบน iPhone, Android, และ iPad
  6. ไม่มี Critical Bug ใดๆ ในส่วน Payment Flow
